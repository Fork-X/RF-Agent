# 从零搭建个人 Agent —— RF-Agent 技术分享

> 📅 一份面向开发者的 Agent 系统构建思路分享
> 重点：设计思路 > 实现细节 | 适合 30 分钟技术分享

---

## 第一部分：什么是 Agent？为什么要从零搭建？

### Agent 和普通 ChatBot 有什么区别？

普通 ChatBot 就像一个"问答机器"——你问它一个问题，它给你一个回答，就完事了。

而 Agent 更像一个**能干活的助手**。它不仅能"说"，还能"做"：
- **自主决策**：面对任务，它能自己判断"先做什么、再做什么"
- **使用工具**：它能读写文件、执行命令、搜索网页——就像一个会用电脑的人
- **多步完成复杂任务**：比如"帮我重构这个函数"，它会自己读代码、分析问题、写新代码、验证结果

一句话总结：ChatBot 只能"对话"，Agent 能"做事"。

### 为什么不用现成框架？

市面上有不少成熟框架——AutoGen、LangGraph、CrewAI。它们功能强大，但对于个人项目来说：

1. **理解底层原理**：从零搭建，每一行逻辑你都清楚。出了 bug，你知道去哪里找
2. **完全掌控**：不受框架设计哲学的约束，想怎么改就怎么改
3. **轻量级**：不引入一堆你用不到的依赖，项目干净清爽
4. **无黑箱**：框架内部的状态管理、消息传递都是透明的

### RF-Agent 解决了什么问题？

RF-Agent（小枫）是一个**私人助理系统**，它能处理多种任务：
- 💻 **代码任务**：读文件、写代码、搜索代码、执行命令
- 🎵 **日常任务**：播放音乐、打开应用、搜索网页
- 💬 **闲聊问答**：直接对话回复，不需要调用任何工具

## 第二部分：系统全景图

### 2.1 整体架构图

```
用户 CLI（命令行交互界面）
  │
  ▼
主 Agent "小枫"（路由决策层）
  │    ├─ Memory（记忆系统：Core + Life 双层）
  │    └─ Skill（路由层策略：tags=["router"]）
  │
  ├──────────────┬──────────────┐
  ▼              ▼              ▼
Coding Agent   Daily Agent    直接回复
(代码任务)     (日常任务)     (闲聊/问答)
  │              │
  ▼              ▼
代码工具集      日常工具集
read_file       open_app
write_file      run_shell
edit_file       browser
search_code     tavily_search
execute_command open_url
  │              │
  ▼              ▼
编码 Skill      执行层 Skill
tags=["coding"] tags=["daily"]
```

主 Agent 就像一个**项目经理**——它自己不干活，但它知道该把什么任务分给谁。Coding Agent 和 Daily Agent 就是两个"专业工程师"，各自带着自己的工具箱和操作手册。

### 2.2 一次完整请求的旅程

这是系统中最重要的流程——以用户说"帮我播放周杰伦的歌"为例：

```
用户："帮我播放周杰伦的歌"
  │
  ▼
① Skill 触发检测
   遍历所有 router 层 Skill
   "播放" 命中 music-player Skill → 注入路由指引到消息历史
  │
  ▼
② 主 Agent ReAct 循环
   LLM 阅读路由指引 → 看到"音乐需求应路由到 Daily Agent"
   决定调用 daily_agent Tool
   参数：task = "帮我播放周杰伦的歌"（保留原话！）
  │
  ▼
③ 创建临时 Daily Agent（无状态新实例）
   加载 daily 标签的 Skill
   带上日常工具集（run_shell、open_app 等）
  │
  ▼
④ Daily Agent ReAct 循环
   "播放" 命中 netease-music-assistant Skill → 注入执行指引
   LLM 按 Skill 指引：先检查环境 → 搜索歌曲 → 播放
   调用 run_shell 工具执行 ncm-cli 命令
  │
  ▼
⑤ 结果层层返回
   Daily Agent 执行结果 → 主 Agent 收到 Tool 返回值 → 用户看到回复
```

注意第②步的关键设计：主 Agent 传递给子 Agent 的是**用户原话**，不会自作主张改写需求。这避免了"传话走样"的问题。

### 2.3 核心概念速览表

| 概念 | 一句话解释 | 类比 |
|------|-----------|------|
| Agent | 能自主决策和使用工具的 LLM 应用 | 项目经理——接到任务后自己规划和执行 |
| Tool | Agent 可以调用的原子操作函数 | 工具箱里的锤子、螺丝刀——每个干一件事 |
| Skill | 用 Markdown 写的策略指引，注入 LLM 上下文 | 操作手册/SOP——告诉你该怎么做 |
| Memory | Agent 的短期和长期记忆 | 笔记本（短期）+ 大脑（长期） |
| ReAct 循环 | 思考→行动→观察→再思考的循环 | "想一想，试一试，看结果，再想想" |
| Session | 会话的自动存档和恢复机制 | 游戏存档——关掉再开，进度还在 |

## 第三部分：六大核心模块详解

### 3.1 Agent 核心引擎（心脏）

🔍 **背景&问题**

LLM 本质上只是一个"输入文本→输出文本"的函数，它不能自己决定下一步做什么，更不能调用外部工具。如果我们只是简单地调用 API 拿到回复，那它永远只是一个聊天机器人。问题在于：**如何让一个只会"说话"的模型变成一个能"做事"的助手？** 答案是给它套上一个循环引擎——ReAct 循环。

🎯 **模块定位**：驱动 LLM "思考-行动" 循环的引擎——Agent 一切行为的发动机。

🏗️ **设计思路**

Agent 就像一个不断"思考→行动→观察结果→再思考"的循环机器：

1. 把对话历史打包发给 LLM
2. LLM 返回结果——要么是"我要用某个工具"，要么是"我的最终回答"
3. 如果要用工具→执行工具→把结果放回历史→回到第 1 步
4. 如果是最终回答→循环结束，返回给用户

📊 **流程图**

```
用户消息 ──→ 加入消息历史 ──→ ┌─────────────────────────────┐
                              │    ReAct 循环（最多 N 轮）    │
                              │                               │
                              │  发送历史给 LLM               │
                              │       │                       │
                              │  LLM 返回结果                 │
                              │       │                       │
                              │  ┌────┴────┐                  │
                              │  │有工具调用?│                  │
                              │  └────┬────┘                  │
                              │   是  │  否                    │
                              │   ▼   └──→ 返回文本响应 ──→ 结束│
                              │  执行工具                      │
                              │  结果入历史                    │
                              │  继续循环 ↑                    │
                              └─────────────────────────────┘
```

💡 **关键设计决策**

1. **错误不崩溃**：工具执行失败时，错误信息作为文本返回给 LLM，LLM 可以自己决定重试还是换方案。就像告诉项目经理"这条路走不通"，他会自己想别的办法——而不是整个项目停摆。

2. **中断控制**：支持用户随时按 ESC 打断循环。通过 `asyncio.Event` 实现优雅中断，不会丢失已有的对话上下文。

📝 **代码一瞥**：ReAct 循环核心（精简版）

```python
async def run(self, user_message: str) -> str:
    self._messages.append({"role": "user", "content": user_message})
    while iteration < self._max_iterations:
        result = await self._client.agent_call(messages, tools)
        if result.tool_calls:
            for tc in result.tool_calls:
                tool_result = await self._execute_tool_call(tc)
                self._messages.append(tool_result)
            continue
        return result.message.get("content", "")
```

### 3.2 Tool 系统（工具箱）

🔍 **背景&问题**

LLM 被"关在"语言空间里——它能写出"读取 config.yaml 文件"这句话，但它真的没法去读那个文件。即使它知道答案在文件里，也只能凭记忆（训练数据）猜测。**如何让 LLM 突破"只能说不能做"的限制？** 需要一套标准化的工具接口，让 LLM 的"意图"变成真实的"操作"。

🎯 **模块定位**：赋予 Agent "动手能力"的工具注册和执行系统。

🏗️ **设计思路**

每个 Tool 有三样东西：
- **说明书**（ToolDefinition）：告诉 LLM 这工具干嘛的、怎么用
- **实际功能**（implementation）：真正执行的代码
- **权限标记**（readonly）：是否只读，用于安全控制

工厂函数像"组装工具箱"——Coding Agent 拿代码工具箱（文件读写、代码搜索），Daily Agent 拿日常工具箱（打开应用、执行命令）。LLM 只看得到说明书，根据说明书决定调什么工具、传什么参数。

📊 **Tool 调用的完整生命周期**

```
Agent 启动
    │
    ▼
工厂函数组装工具集
    create_filesystem_tools() → [read_file, write_file, edit_file, list_dir]
    create_code_tools()       → [search_code, verify_code]
    create_command_tools()    → [execute_command]（注入 work_dir 安全边界）
    │
    ▼
注册到 Agent 内部工具表 { name → (定义, 实现, 只读标记) }
    │
    ▼ （运行时）
LLM 收到所有工具的"说明书"列表
    │
    ▼
LLM 决定调用某个工具 → 生成 ToolCall{name, arguments}
    │
    ▼
Agent 查表 → 找到实现函数 → 执行 → 返回 ToolResult{content}
    │
    ▼
结果加入消息历史 → LLM 下一轮继续思考
```

💡 **关键设计决策**

1. **三元组模式**：`(定义, 实现, 只读标记)` 分离——同一个工具可以在不同模式下有不同权限。比如 Plan 模式只允许使用 `readonly=True` 的工具，防止误操作。

2. **参数化工厂**：`execute_command` 通过闭包注入 `work_dir`——不同 Agent 有不同的安全边界。Coding Agent 只能在项目目录里执行命令，越界操作会被拦截。

📊 **安全机制流程图**

```
execute_command 接收命令
    │
    ├─ ① 硬黑名单 → sudo/shutdown/reboot → 🚫 直接拒绝
    │
    ├─ ② 危险命令检测 → rm/mv/cp 操作 → 检查路径是否越界
    │       │
    │       ├─ 路径在 work_dir 内 → ✅ 允许执行
    │       └─ 路径越界 → ③ 弹出确认 → 用户 y/N
    │
    └─ 普通命令 → ✅ 直接执行
```

📝 **代码一瞥**：错误隔离——Tool 失败不崩溃

```python
async def _execute_tool_call(self, tool_call):
    try:
        result = entry.implementation(args)
        if inspect.isawaitable(result):
            result = await result
        return ToolResult(content=str(result))
    except Exception as e:
        return ToolResult(content=f"工具执行失败: {e}")  # 异常→文本，不中断循环
```

### 3.3 Skill 系统（操作手册）

🔍 **背景&问题**

给 Agent 加了工具，它能做事了——但它不知道**该怎么做**。比如你给它一把锤子，它可能用锤子去拧螺丝。现实场景中，同样是"播放音乐"，不同平台有不同的操作步骤、不同的命令格式。每次都靠 LLM 自己摸索，效率低还容易出错。**如何低成本地给 Agent 注入"操作经验"？** 不是靠写更多代码，而是写一份操作手册。

🎯 **模块定位**：用 Markdown 文件定义 Agent 的"行为策略"——不改代码就能改变 Agent 行为。

🏗️ **设计思路**

Tool 给 Agent "能做什么"的能力，Skill 告诉 Agent "该怎么做"的策略。这两者的关系就像：

- Tool = 工人手里的工具（锤子、螺丝刀）
- Skill = 操作手册（"先检查尺寸→再钉钉子→最后打磨"）

一个 Skill 就是一个 `SKILL.md` 文件，用 YAML 头部声明触发条件，Markdown 正文写操作指引。当用户说了某个关键词（如"播放"），对应的 Skill 被激活，其操作指引被注入到 LLM 的上下文中。LLM 阅读指引后，就像拿到了一本操作手册，按手册步骤行事。

📊 **Skill 的触发和激活流程图**

```
用户说："帮我播放音乐"
    │
    ▼
遍历所有已加载的 Skill
    │
    ├─ music-player Skill：triggers 含 "播放" → ✅ 命中！
    ├─ python-code-review Skill：无匹配触发词 → ❌ 跳过
    └─ web-search Skill：无匹配触发词 → ❌ 跳过
    │
    ▼
按 (优先级, 匹配分数) 排序
    music-player: priority=10, score=1 → 排在最前
    │
    ▼
激活 Skill → 将 Markdown 正文作为 system 消息注入消息历史
    │
    ▼
LLM 看到："[Skill 上下文 - music-player]
           你是音乐播放路由中心...
           网易云相关 → 调用 daily_agent
           QQ音乐 → 回复开发中"
    │
    ▼
LLM 据此做出路由决策
```

📊 **分层设计图（最关键的架构决策之一）**

```
┌─────────────────────────────────────────────────────┐
│                    Skill 分层架构                      │
│                                                       │
│  ┌──────────────── 路由层 ────────────────┐           │
│  │  主 Agent 加载 (skill_tags=["router"]) │           │
│  │                                        │           │
│  │  music-player (priority=10)            │           │
│  │    → "音乐需求路由到 Daily Agent"       │           │
│  │  python-code-review                    │           │
│  │    → "代码审查路由到 Coding Agent"      │           │
│  └────────────────────────────────────────┘           │
│                     │ 路由转发                          │
│                     ▼                                  │
│  ┌──────────────── 执行层 ────────────────┐           │
│  │  子 Agent 加载 (skill_tags=["daily"])  │           │
│  │                                        │           │
│  │  netease-music-assistant (priority=5)  │           │
│  │    → 7步完整音乐执行流程               │           │
│  │  netease-music-cli                     │           │
│  │    → CLI 工具具体用法指引              │           │
│  └────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────┘
```

💡 **关键设计决策**

1. **路由层 vs 执行层分离**：路由层 Skill 只告诉主 Agent "该找谁做"，执行层 Skill 才告诉子 Agent "具体怎么做"。这样添加新能力（如 QQ 音乐支持）只需加一条路由规则，不影响已有的执行逻辑。

2. **策略外化为 Markdown**：修改 Skill 不需要改代码、不需要重启。对于快速迭代的个人项目，这极大降低了维护成本——改一个 `.md` 文件比改代码安全得多。

📝 **代码一瞥**：SKILL.md 文件格式

```yaml
---
name: music-player
description: 音乐播放路由 Skill
priority: 10
tags: [router]
triggers: [播放, 音乐, 歌]
tools: [daily_agent]
---
# 音乐播放指引
当用户提出音乐相关需求时，路由到 Daily Agent ...
```

### 3.4 Memory 系统（记忆）

🔍 **背景&问题**

LLM 没有真正的"记忆"——每次对话都是从零开始。你昨天告诉它"我喜欢用 VS Code"，今天它就忘了。就算在同一次对话中，上下文压缩后早期的细节也会丢失。**如何让 Agent 像人一样"记住"重要信息？** 需要在 LLM 之外建立独立的记忆存储。

🎯 **模块定位**：让 Agent 拥有短期和长期记忆，不会"忘记"重要信息。

🏗️ **设计思路**

人的记忆分短期（今天聊了什么）和长期（你喜欢什么），Agent 也是：

- **Core Memory**（长期记忆）：JSON 文件，存储用户偏好、重要决策等。每条记忆带"强化计数器"——出现次数越多越重要，就像你反复提到某件事，大脑就记得更牢
- **Life Memory**（短期记忆）：Markdown 文件，每次对话上下文压缩时自动保存摘要。就像每天写的日记
- **记忆整合**：定期分析近 14 天的短期记忆，把反复出现的主题沉淀为长期记忆——就像你回顾一周日记，发现"这个人每天都在调试音乐功能"，于是记住"他在做音乐相关开发"

📊 **Memory 系统流程图**

```
                     日常对话进行中
                          │
                          ▼
                  上下文过长，触发压缩
                          │
              ┌───────────┴───────────┐
              ▼                       ▼
      LLM 生成对话摘要          异步保存 Life Memory
      旧消息标记归档            lifeMemory-主题-日期.md
              │                       │
              ▼                       ▼
      摘要节点插入消息历史       每 3 次压缩触发整合
                                      │
                                      ▼
                               ┌──────────────┐
                               │ LLM 分析最近  │
                               │ 14天 Life     │
                               │ Memory        │
                               │      │        │
                               │  识别高频主题  │
                               │      │        │
                               │  更新 Core    │
                               │  Memory       │
                               │ (强化计数+1)  │
                               └──────────────┘
```

```
    recall_memory 工具：用户或 LLM 主动检索记忆
          │
          ├─ 搜索 Core Memory（关键词匹配，找长期记忆）
          └─ 列出最近 7 天 Life Memory（找近期对话摘要）
          │
          ▼
      拼接结果返回给 LLM → LLM 基于记忆回答
```

💡 **关键设计决策**

1. **双层分离**：Core Memory（精炼、结构化）和 Life Memory（原始、时间线）各司其职。不是所有信息都值得长期记住，双层设计让系统在"记太多"和"记太少"之间找到平衡。

2. **强化计数器**：`reinforce_count` 机制让重要记忆自然浮现——出现 10 次的偏好比出现 1 次的更可信，这比简单的"记/不记"更符合人类记忆规律。

### 3.5 Context 管理（上下文窗口）

🔍 **背景&问题**

LLM 的上下文窗口不是"越大越好用"。以 168K token 的上下文窗口为例，研究表明大约用到 40% 就开始"走下坡路"——信息越多，模型反而越难找到重点，输出质量下降。更现实的问题是：一次包含代码分析的对话，十几轮下来轻松超过 token 上限。**如何在有限的窗口内，让 LLM 始终"看到"最有价值的信息？**

🎯 **模块定位**：在有限的 Token 窗口内，保留最有价值的对话信息。

🏗️ **设计思路**

LLM 有 Token 上限（比如 16000），长对话必须"裁剪"历史。就像你的桌面空间有限，必须定期整理——把不常用的文件归档，只留手边需要的。

策略很简单：**保留最近 3 轮对话（6 条消息），更早的内容由 LLM 生成摘要**。用内部元数据标记（`_archived`、`_summary`）管理消息生命周期。

📊 **上下文压缩流程图**

```
消息历史（压缩前）：
┌─────────────────────────────────────────────────────┐
│ [system] 系统提示词                                    │
│ [system] Skill 上下文                                  │
│ [user] 消息1  ─┐                                      │
│ [assistant] 回复1 │                                    │
│ [user] 消息2      │── 旧消息（待压缩）                  │
│ [tool] 工具结果   │                                    │
│ [assistant] 回复2─┘                                    │
│ [user] 消息3  ─┐                                      │
│ [assistant] 回复3 │── 最近 6 条（保留）                 │
│ [user] 消息4      │                                    │
│ [tool] 工具结果   │                                    │
│ [assistant] 回复4 │                                    │
│ [user] 消息5  ─┘                                      │
└─────────────────────────────────────────────────────┘
                            │ 触发压缩
                            ▼
消息历史（压缩后）：
┌─────────────────────────────────────────────────────┐
│ [system] 系统提示词                                    │
│ [system] Skill 上下文                                  │
│ [user] 消息1     _archived=True（保留但不发送给 LLM）  │
│ [assistant] 回复1 _archived=True                       │
│ ...                                                    │
│ [system] "[历史摘要] 之前讨论了..."  _summary=True ←新增│
│ [user] 消息3  ─┐                                      │
│ [assistant] 回复3 │── 最近 6 条（继续保留）             │
│ ...             ─┘                                     │
└─────────────────────────────────────────────────────┘

发送给 LLM 的内容 = system消息 + 摘要节点 + 最近消息
（_archived 的不发送，_ 开头的字段在发送前全部清理）
```

📊 **压缩触发条件**

压缩不是按消息条数触发，而是基于 **Token 使用量**。LLM 每次返回都会带上 `token_usage`，当 `total_tokens >= 16000` 时自动触发压缩。这比数消息数量更精确——一条含大段代码的消息可能比十条闲聊消耗更多 Token。

```
LLM 返回结果
  │
  ▼
检查 token_usage.total_tokens
  │
  ├─ < 16000 → 正常继续
  └─ >= 16000 → 触发 compress_context()
       │
       ├─ 保留最近 6 条消息
       ├─ 旧消息交给 LLM 生成摘要
       ├─ 旧消息标记 _archived=True
       └─ 插入 _summary 摘要节点
```

📊 **压缩后 LLM 上下文组装**（`_get_llm_messages`）

压缩完成后，下一轮发给 LLM 的内容不是全部消息历史，而是经过精心组装的：

```
_get_llm_messages() 组装逻辑：

遍历 _messages 列表
  │
  ▼
查找最后一条 _summary 节点
  │
  ├─ 找到 → 从该节点开始构建上下文
  │         │
  │         ├─ [system] 系统提示词
  │         ├─ [system] 摘要节点（"之前讨论了..."）
  │         ├─ [user/assistant] 摘要之后的消息
  │         └─ 发送前：剥离所有 _ 开头的元数据字段
  │
  └─ 未找到 → 发送全部非 _archived 消息

结果：LLM 看到的是 = system提示 + 历史摘要 + 最近对话
     （_archived、_summary 等内部标记对 LLM 完全不可见）
```

关键细节：所有以 `_` 开头的字段（`_archived`、`_summary` 等）是给系统内部用的元数据，发送给 LLM 之前会全部剥离——就像你整理好的笔记给同事看时，会撕掉上面的便利贴标记。

📊 **压缩-记忆联动**

压缩不只是"丢掉旧消息"，它还会异步触发记忆保存：

```
compress_context() 执行压缩
  │
  ├─ ① 生成摘要 + 标记归档（主流程）
  │
  └─ ② 异步触发 update_life_memory_async()（fire-and-forget）
       │    不阻塞主流程，用户无感知
       │
       ├─ 将活跃消息的摘要保存为 Life Memory
       │  （lifeMemory-主题-日期.md）
       │
       └─ 检查压缩计数器
            │
            ├─ 未满 3 次 → 结束
            └─ 每 3 次压缩 → 触发 Core Memory 整合
                 │
                 └─ 分析近期 Life Memory → 沉淀为长期记忆
```

💡 **关键设计决策**

1. **归档而非删除**：旧消息标记 `_archived=True` 而不是直接删除——会话持久化时仍然保留完整历史，方便回溯和调试。

2. **摘要节点机制**：`_summary=True` 标记的系统消息作为"历史锚点"，让 LLM 知道"之前发生过什么"，保持对话连贯性。

3. **Fire-and-Forget 异步记忆**：压缩时的 Life Memory 保存采用"发射后不管"模式——即使保存失败也不影响对话主流程。对话体验 > 记忆完整性。

### 3.6 会话持久化（存档系统）

🔍 **背景&问题**

Agent 的所有对话状态都在内存中——程序一关，一切归零。你花了半小时和 Agent 调试一个 bug，中间关了终端，再打开就要从头开始解释"我们刚才在做什么"。**如何让 Agent 的对话跨越程序生命周期？** 需要一个像游戏存档一样的持久化机制。

🎯 **模块定位**：让 Agent 的对话不会因为程序退出而丢失——关上笔记本再打开，上次写到哪里还在那里。

🏗️ **设计思路**

Session Store 就像游戏的**存档/读档**系统：

- 每次对话后自动保存（自动存档），程序启动时自动恢复（读档）
- `/clear` 命令 = 归档旧存档开始新游戏
- 按工作目录隔离：不同项目目录有不同的存档文件——就像不同游戏各有各的存档槽

📊 **保存策略**

```
对话进行中
  │
  ▼
每轮对话结束 → save_session()
  │
  ├─ 过滤：移除 system 消息（除摘要节点）
  ├─ 保留：_archived 和 _summary 标记
  └─ 写入：.sessions/{项目名}-{目录哈希}.json
```

📊 **恢复流程**

```
程序启动
  │
  ▼
load_session() → 读取 JSON
  │
  ├─ 文件不存在 → 全新对话
  ├─ JSON 损坏 → 容错返回空（不崩溃）
  └─ 正常加载 → agent.load_messages()
       │
       ▼
  内存中 _messages = [system_prompt] + [历史消息]
       │
       ▼
  包含 _summary 节点？
  ├─ 是 → _get_llm_messages() 自动从摘要开始（LLM 继承历史背景）
  └─ 否 → 正常加载所有消息
       │
       ▼
  显示："已恢复上次会话（N 轮对话）"
```

📊 **归档机制**

```
用户执行 /clear
  │
  ├─ ① 清空内存：agent.clear_history()
  ├─ ② 归档文件：原 JSON 重命名为 {name}-archive-{时间戳}.json
  └─ ③ 显示："📦 旧对话已归档"
  │
  ▼
全新对话开始（干净的 _messages）
```

💡 **关键设计决策**

1. **归档而非删除**：`/clear` 不会删掉旧对话，而是重命名为带时间戳的归档文件。这样既有了新的开始，又不丢失历史——就像游戏的存档列表。

2. **system 消息不持久化**：保存时过滤掉 system 消息（Skill 注入的策略、系统提示等），恢复时依赖 Agent 初始化重新生成。这避免了恢复时出现过时的系统提示——就像每次开机重新加载驱动，而不是用上次的缓存。

📊 **压缩 + 持久化协作全景**

```
                   压缩 + 持久化 协作全景

对话进行中 ──→ token 超限 ──→ 触发压缩
                              │
                ┌─────────────┼─────────────┐
                ▼             ▼             ▼
         旧消息标记      生成摘要节点    异步保存
         _archived      _summary       Life Memory
                              │
                              ▼
                     save_session() 保存到 JSON
                     （包含 _archived + _summary 标记）
                              │
                              ▼
                     下次启动 load_session()
                     （恢复完整状态，含归档标记和摘要）
                              │
                              ▼
                     _get_llm_messages() 从摘要开始构建
                     （LLM 无感知地继承历史上下文）
```

## 第四部分：从零搭建的学习路径

```
Step 6: 安全与可靠性 ──── 工具权限、命令安全、错误隔离、中断控制
    ▲
Step 5: Memory 系统 ───── 双层记忆、上下文压缩、会话持久化
    ▲
Step 4: 多 Agent 协作 ── Agent 工厂、主Agent路由、skill_tags 分层
    ▲
Step 3: Skill 系统 ───── SKILL.md 格式、解析器、触发匹配、策略注入
    ▲
Step 2: Tool 系统 ────── 工具定义格式、注册机制、工厂函数、安全检查
    ▲
Step 1: 最小 Agent ───── ReAct 循环、消息历史、LLM 调用、文本返回
```

### Step 1：最小 Agent

**目标**：实现一个能和 LLM 对话、能调用工具的最小循环。

**关键产出**：`Agent` 类 + `run()` 方法 + 消息历史管理。

**建议**：先不加任何工具，确保"用户输入→LLM回复→显示"这条链路跑通，再加工具支持。

### Step 2：Tool 系统

**目标**：定义统一的工具格式，让 LLM 能通过 Function Calling 调用真实功能。

**关键产出**：ToolDefinition 格式、register_tool 方法、execute_tool_call 方法。

**建议**：从 `read_file` 这样的简单只读工具开始，验证"LLM 决定调用→执行→结果回传"整条链路。

### Step 3：Skill 系统

**目标**：通过声明式 Markdown 文件注入策略，不写代码就能扩展 Agent 能力。

**关键产出**：SKILL.md 格式定义、SkillParser（YAML frontmatter 解析）、SkillLoader（递归扫描+按 tags 过滤）、触发词匹配和 Skill 注入机制。

**建议**：先实现手动激活（`activate_skill` 工具），再实现自动触发。Skill 系统的核心价值是把策略从代码中抽出来。

### Step 4：多 Agent 协作

**目标**：实现主 Agent 路由 + 子 Agent 执行的分工模式。

**关键产出**：Agent 工厂函数、子 Agent 包装为 Tool、`skill_tags` 分层加载。

**建议**：先做"主 Agent + 1 个子 Agent"，验证路由和返回值传递，再加第二个子 Agent。

### Step 5：Memory 系统

**目标**：让 Agent 有"记忆"，长对话不丢失上下文。

**关键产出**：Core Memory（JSON 长期存储）、Life Memory（Markdown 日常摘要）、上下文压缩、会话持久化（session JSON）、记忆整合（consolidate）。

**建议**：先实现上下文压缩（最紧急的痛点），再加 Memory 存储和检索。

### Step 6：安全与可靠性

**目标**：让 Agent 在日常使用中不"闯祸"。

**关键产出**：工具 readonly 标记、命令执行三层防护（黑名单+路径检测+用户确认）、错误隔离（try/except 转文本）、中断控制（asyncio.Event）、迭代上限（max_iterations）。

**建议**：安全机制从第一天就要考虑，但可以渐进式增强。至少 `max_iterations` 和错误隔离要在 Step 1 就加上。

## 第五部分：项目不足与改进方向

### 🏗️ 架构层面

1. **子 Agent 无状态代价**：每次调用创建新实例，重复文件读取和项目理解。改进方向：会话级 Agent 复用 + 共享文件缓存
2. **单向通信局限**：子 Agent 不能反向调用主 Agent，需要澄清需求时只能猜测。改进方向：引入消息总线或事件系统
3. **串行任务执行**：多个独立 Tool 无法并行调用。改进方向：`asyncio.gather` 并行执行无依赖的工具调用
4. **缺乏任务编排**：没有 DAG 式依赖管理，复杂任务只能线性执行。改进方向：轻量级任务图 + 条件分支

### 🔧 Tool 系统

5. **import 位置不规范**：部分文件的 import 语句位置违反 PEP 8，虽不影响运行但不利于维护
6. **工具结果无结构化**：所有 Tool 返回纯字符串，无法被程序化解析。改进方向：JSON + 人类可读文本混合格式
7. **无工具版本管理**：Tool 定义变更时无法追踪兼容性，旧会话恢复可能出问题

### 🧩 Skill 系统

8. **触发词匹配过于简单**：纯关键词/子串匹配，"帮我来首歌"无法匹配触发词"播放"。改进方向：轻量级意图分类器或 embedding 语义匹配
9. **Skill 间无显式依赖**：路由层和执行层 Skill 的关联是隐式的。改进方向：frontmatter 增加 `depends_on` 字段
10. **无 Skill 生命周期管理**：Skill 一旦激活就持续存在，无法自动停用。改进方向：引入作用域（per-message / per-session）

### 🧠 Memory 系统

11. **检索是简单词级匹配**：O(n) 遍历所有记忆做子串匹配，无语义理解。改进方向：向量化存储 + 语义搜索
12. **Life Memory 无自动清理**：文件只增不减，长期运行后目录无限增长。改进方向：超过 N 天自动归档
13. **Core Memory 整合依赖 LLM 输出 JSON**：解析不稳定，LLM 可能输出格式不一致。改进方向：使用 structured output

### 📐 Context 管理

14. **压缩策略不区分内容类型**：代码和普通文本用同样的策略压缩，代码信息可能丢失。改进方向：选择性压缩
15. **摘要质量无验证**：压缩后无法确认关键信息（文件路径、已做修改）是否保留。改进方向：关键信息检查点

### 🆚 与主流框架的差距

| 对比维度 | RF-Agent | AutoGen | LangGraph | CrewAI |
|----------|----------|---------|-----------|--------|
| Agent 间通信 | 单向调用 | ✅ 多轮对话 | ✅ 状态图 | ✅ 消息总线 |
| 流程控制 | 线性 ReAct | 对话驱动 | ✅ 条件分支+循环 | 任务驱动 |
| 并行执行 | ❌ 串行 | ❌ | ✅ 并行节点 | ✅ 并行任务 |
| 任务依赖 | ❌ 无 | ❌ | ✅ 边和条件 | ✅ DAG 编排 |
| 轻量级 | ✅ 极简依赖 | 中等 | 中等 | 中等 |

RF-Agent 的优势在于**极简、可控、易理解**——没有框架黑箱，每一行逻辑都透明。

---

## 结语

构建个人 Agent 不需要复杂的框架。

一个 **ReAct 循环** + **Tool 注册表** + **Skill 注入** 就能跑起来。随着需求增长，再逐步加入 Memory、Context 压缩、安全机制和多 Agent 协作。

从零搭建的核心价值不在于造出一个"更好的框架"，而在于**理解每一层解决了什么问题**：
- Agent 引擎解决了"如何让 LLM 自主行动"
- Tool 系统解决了"如何让 LLM 与真实世界交互"
- Skill 系统解决了"如何低成本地扩展和调整 Agent 行为"
- Memory 系统解决了"如何让 Agent 不忘记重要信息"
- Context 管理解决了"如何在有限窗口内保留最有价值的信息"
- 会话持久化解决了"如何让 Agent 的记忆跨越程序重启"

当你理解了这些，再去看任何 Agent 框架，都不会觉得是黑箱了。
